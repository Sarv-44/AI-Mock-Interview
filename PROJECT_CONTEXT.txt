PURPOSE
This repository is a FastAPI-based interview preparation product with a server-rendered frontend.
It supports:
- topic-based mock interviews
- role-based mock interviews
- custom interview templates created by signed-in users
- study plans / prep paths tied to roles
- public and personalized score leaderboards
- transcript analysis from uploaded audio using Whisper
- PDF export of interview reports
- topic ratings, activity tracking, profile history, and admin catalog management

If I need to resume work later, this file should be treated as the first-stop project summary.
The actual runtime source of truth is the Python code, especially backend/main.py and backend/database.py.


HIGH-LEVEL ARCHITECTURE
Backend:
- FastAPI app in backend/main.py
- Business logic and persistence in backend/database.py
- Transcript scoring and feedback heuristics in backend/analytics.py
- Auth token creation/verification in backend/auth.py
- Catalog defaults and question-bank parsing in backend/interview_catalog.py
- PDF report generation in backend/pdf_generator.py

Frontend:
- HTML pages in templates/
- CSS in static/css/
- Browser logic in static/js/
- The frontend is not React at runtime; the old React path was removed
- The current visual direction is a shared dark editorial UI system with premium glass surfaces, tighter spacing, smaller contextual action bars, and stronger typography instead of oversized generic dashboard styling

Data layer:
- MySQL via mysql-connector-python
- Database auto-initialization happens on app startup through init_database()
- Many records store structured payloads in JSON columns or JSON strings

Audio / AI:
- openai-whisper is used locally for transcription
- Transcript quality is then analyzed with in-repo heuristics, not with a remote API

PRODUCT SURFACE
1. Anonymous or signed-in users can run interview sessions.
2. Signed-in users can save history, download PDFs, rate topics, create custom interviews, and create study plans.
3. Admin users can manage topics, questions, and roles from the admin UI.
4. Study plans generate multi-step prep paths and study materials for a selected role.
5. Completing a study-plan-backed interview automatically updates the linked study step progress.
6. Leaderboards expose anonymized ranking for everyone, while signed-in users can also see their own real placement.

LEADERBOARD SYSTEM
This is the current MVP leaderboard implementation.

Backend support:
- backend/database.py:get_score_leaderboard()
- backend/main.py: GET /api/leaderboards
- backend/main.py: GET /leaderboards

Supported boards:
- overall
- weekly
- topic

Current leaderboard rules:
- overall board ranks signed-in users by AVG(final_score) across all saved sessions
- weekly board ranks signed-in users by AVG(final_score) across the last 7 days
- topic board ranks signed-in users by AVG(final_score) for pure topic-mode sessions on one selected topic
- guest sessions are excluded because they have no stable user_id
- everyone except the currently signed-in user is anonymized as User_<hash>

Minimum eligibility thresholds:
- overall: 3 saved interviews
- weekly: 2 saved interviews in the last 7 days
- topic: 2 saved topic-mode interviews on the selected topic

Response shape from /api/leaderboards:
- success
- board
- title
- entries
- current_user_entry
- current_user_progress
- minimum_interviews
- timeframe_days
- total_ranked_users
- topic

Frontend for this module:
- template: templates/leaderboards.html
- script: static/js/leaderboards.js
- style: static/css/leaderboards.css

Frontend responsibilities:
- switch between overall, weekly, and topic boards
- load topics from the interview catalog for topic selection
- render public leaderboard rows
- show the signed-in user's exact rank when eligible
- show eligibility progress when the user is signed in but not yet ranked


AUTH MODEL
Auth is custom and lightweight.
It is NOT JWT.

How it works:
- backend/auth.py creates a signed token made of base64url(payload) + HMAC signature
- payload includes user_id, email, username, is_admin, and exp
- verify_auth_token() checks signature and expiration
- extract_bearer_token() expects Authorization: Bearer <token>

Access rules:
- user profile and history routes are restricted to the matching signed-in user
- admin routes require a signed-in user whose DB row has is_admin = true
- sessions belonging to guests are effectively public because there is no owning user_id
- sessions linked to a user require the matching authenticated user


DATABASE OVERVIEW
Most important note:
- The live schema is best understood from backend/database.py:init_database()
- SQL files are useful references, but the runtime initializer is more authoritative because it also performs incremental ALTER checks and seeds defaults

Schema files:
- DATABASE_SCHEMA.sql is the consolidated schema snapshot for the current app

Core tables created by init_database():
- users
- interviews
- topic_ratings
- topic_activity
- topics
- questions
- job_roles
- custom_interviews
- topic_study_materials
- study_plans
- study_plan_steps

Table purposes:
- users: local account store with salted PBKDF2 password hashes and is_admin flag
- interviews: one row per saved interview session, with complete_data JSON as the main payload
- topic_ratings: user feedback per interview session
- topic_activity: tracks topic interview activity, mainly for signed-in topic sessions
- topics: active interview topics shown in catalog
- questions: question bank linked to topics
- job_roles: role definitions plus topic weights and duration defaults
- custom_interviews: saved user-authored interview templates
- topic_study_materials: overview, revision notes, mistakes, rapid fire points, and prompts for planner topics
- study_plans: saved prep path header row
- study_plan_steps: per-step schedule and snapshot/progress data for a plan

Important relationships:
- interviews.user_id -> users.user_id
- topic_ratings.session_id -> interviews.session_id
- topic_ratings.user_id -> users.user_id
- topic_activity.session_id -> interviews.session_id
- topic_activity.user_id -> users.user_id
- questions.topic_id -> topics.topic_id
- custom_interviews.user_id -> users.user_id
- topic_study_materials.topic_id -> topics.topic_id
- study_plans.user_id -> users.user_id
- study_plan_steps.plan_id -> study_plans.plan_id

INTERVIEW SESSION FLOW
There are several interview modes:
- topic
- role
- custom
- study-plan-derived runs that resolve into topic, role, or mixed plans

General planning behavior:
- GET /api/interview/session-plan builds a topic or role interview plan
- topic mode selects questions for a single topic
- role mode selects questions by role topic weights and duration blueprint

Custom interview behavior:
- Users can define their own questions with weight, target_seconds, and section_label
- Templates are stored in custom_interviews.questions_json
- A template can be converted into a runnable session plan through /api/custom-interviews/{template_id}/session-plan

Study plan behavior:
- A study plan is generated for a selected role and target_days
- The outline is divided into phases like foundation, checkpoints, and simulation
- Steps may be of type:
  - revise
  - topic_round
  - mixed_quiz
  - mock_interview
- Some steps are informational only; revise steps are not directly runnable
- Runnable steps can be turned into interview plans through /api/study-plans/{plan_id}/steps/{step_id}/session-plan

Saving behavior:
- POST /save-interview writes the full session payload into interviews.complete_data
- For study-plan sessions, sync_study_step_result() marks the related step as done and updates snapshot metrics like:
  - last_score
  - best_score
  - attempt_count
  - average_score
  - last_session_id
  - last_completed_at
  - last_confidence
  - last_content_score
  - last_delivery_score
- For topic mode sessions, topic activity is also synced for signed-in users


TRANSCRIPTION AND ANALYTICS
Transcription route:
- POST /transcribe

Input:
- multipart upload with audio file
- form fields such as difficulty, topic, question, sample_answer, ideal_answer, analytics_mode, target_seconds, question_weight

Whisper behavior:
- Loads WHISPER_MODEL first, then WHISPER_FALLBACK_MODEL if needed
- Uses a prompt biased toward technical Indian English / Hinglish
- Tries multiple language hints from WHISPER_LANGUAGE_HINTS
- Normalizes common technical phrases like BFS, DFS, Node.js, React, DBMS, SQL, API, LLM
- Scores candidate transcriptions and accepts the best viable output

Analytics behavior:
- backend/analytics.py is heuristic-heavy and local
- It computes metrics such as:
  - transcript
  - word_count
  - filler words / filler list
  - duration
  - WPM
  - confidence
  - overall score
  - delivery score
  - content score
  - keyword coverage
  - topic relevance
  - intent alignment
  - technical depth
  - strengths
  - improvements
- The module contains topic hints, aliases, signal terms, question guidance, and scoring heuristics


PDF REPORTING
PDF generation lives in backend/pdf_generator.py.

Behavior:
- GET /generate-pdf/{session_id} fetches the saved session, checks access, then builds a PDF
- The PDF includes:
  - session metadata
  - each question and answer
  - highlighted filler words
  - per-question metrics
  - custom interview timing fields when session_mode == custom
  - overall summary averages

Generated files are written into the current working directory unless a custom filename is supplied.
There are existing generated PDFs in the repo root, so this behavior is already visible in the project state.

IMPORTANT IMPLEMENTATION NOTES
1. The runtime schema is more current than the old SQL files.
   If schema confusion happens, read init_database() first.

2. Tokens are custom HMAC tokens, not JWTs.
   Any auth changes should preserve create_auth_token(), verify_auth_token(), and frontend storage expectations.

3. interviews.complete_data is the main session payload store.
   A lot of reporting and history behavior depends on that JSON shape remaining stable.

4. Study-plan result syncing happens automatically on save.
   If save payloads change, make sure study_plan_id, study_step_id, and user_id still reach sync_study_step_result().

5. Whisper loading is local and can fail gracefully.
   If it fails, /transcribe returns an availability error instead of crashing the app.

6. The frontend is page-based and tightly coupled to route names.
   Renaming routes or changing payloads will likely require JS updates.

7. Navigation should stay structured and minimal.
   Avoid turning every page-intro area into a full app sitemap; prefer 2-3 contextual links plus the shared account/header navigation.

8. The visual system should stay coherent across pages.
   Prefer evolving shared primitives in base.css, style.css, and shared page-shell styles before making isolated one-off page tweaks.

9. START.bat has a stray trailing line at the end:
   "icorn backend.main:app --reload"
   It does not appear intentional and is probably leftover noise.

10. Leaderboards currently query on demand.
   There is no snapshot or materialized leaderboard table yet.

11. The repo contains a venv/ directory and generated PDFs.
   Those are not core source files, just environment/runtime artifacts.
